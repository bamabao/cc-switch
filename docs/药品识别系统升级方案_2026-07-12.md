# 爸妈宝 — 药品识别系统升级方案

> 版本：v1.0 | 日期：2026-07-12 | 决策人：王总 | 执行人：刘工

---

## 一、方案概述

### 现状痛点
- 当前药品识别完全依赖火山视觉 API，每次识别都产生费用
- 同一种药反复调用 API，成本浪费
- 网络差时识别失败

### 升级方案
**本地OCR + 火山视觉AI混合识别 + 药品数据库缓存**

```
用户拍照 → 本地OCR（免费） → 识别成功？ → 是 → 查缓存库 → 返回结果
                                ↓ 否
                            火山视觉API（兜底） → 结果写入缓存库 → 返回结果
```

### 核心策略
| 层级 | 技术 | 费用 | 占比预估 |
|------|------|------|----------|
| 第一层 | 本地OCR（免费） | ¥0 | ~80% |
| 第二层 | 火山视觉AI兜底 | 单次约¥0.3 | ~20% |
| 第三层 | 药品数据库缓存 | 查库免费 | 复购零成本 |

## 三、费用估算

### 当前方案（纯火山API）
| 项目 | 单价 | 日均调用 | 月费用 |
|------|------|----------|--------|
| 火山视觉OCR | ~¥0.3/次 | 假设100次 | ~¥900 |

### 升级后方案（混合识别）
| 项目 | 单价 | 日均 | 月费用 |
|------|------|------|--------|
| 本地OCR | ¥0 | ~80次 | ¥0 |
| 火山视觉API（兜底20%） | ~¥0.3/次 | ~20次 | ~¥180 |
| 缓存命中（复购免费） | ¥0 | 逐步增加 | ¥0 |
| **合计** | | | **~¥180** |

### 节省幅度
- 首月预计节省 **~80%**（¥900 → ¥180）
- 随缓存积累，后续月份节省更多
- 日均调用量越大，节省比例越高

## 四、数据库设计

### medicine_cache 表

```sql
CREATE TABLE medicine_cache (
    id INT AUTO_INCREMENT PRIMARY KEY,
    image_hash VARCHAR(64) NOT NULL UNIQUE COMMENT '图片SHA256哈希',
    medicine_name VARCHAR(255) NOT NULL COMMENT '药品名称',
    recognize_result JSON COMMENT '完整识别结果JSON',
    confidence DECIMAL(5,2) COMMENT '识别置信度 0-100',
    source ENUM('local', 'volcano') DEFAULT 'local' COMMENT '识别来源',
    usage_count INT DEFAULT 1 COMMENT '命中次数',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_medicine_name (medicine_name),
    INDEX idx_source (source)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 缓存命中逻辑
1. 用户拍照上传 → 计算图片SHA256
2. 查 medicine_cache.image_hash → 命中直接返回缓存结果
3. 未命中 → 走本地OCR
4. 本地OCR置信度 < 阈值 → 走火山API兜底
5. 识别结果写入缓存，下次同药免费

## 七、风险与预案

| 风险 | 影响 | 预案 |
|------|------|------|
| 本地OCR识别率低 | 频繁触发API，费用增加 | 动态调整置信度阈值 |
| 图片哈希碰撞 | 返回错误药品信息 | 加MD5双哈希校验 |
| 火山API限流/故障 | 识别不可用 | 降级为纯本地OCR+提示 |
| 缓存膨胀 | 查询变慢 | 定期清理低命中率缓存 |
